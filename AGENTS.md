# AGENTS.md

行为准则，用于规避大模型编码常见错误。可根据需要结合项目专属规范使用。

**取舍原则：** 本准则优先保守严谨，而非追求开发速度。简单琐碎的任务可灵活酌情处理。

## 1. 编码前先思考

**不主观臆断、不掩饰疑惑、主动罗列方案取舍。**

动手开发前：
- 明确说出你的所有预设条件，存在不确定就主动提问。
- 若需求存在多种解读方式，全部列出来，不要私自暗自选定一种。
- 如果有更简洁的实现思路，主动说明；合理情况下可以提出异议。
- 遇到模糊不清的地方立刻停手，点明疑惑之处并发起询问。

## 2. 简洁优先

**用最少代码解决问题，不做多余、 speculative 冗余设计。**

- 不额外开发需求之外的功能。
- 只使用一次的代码，不做过度抽象封装。
- 不私自增加未要求的灵活性、可配置性设计。
- 不为不可能发生的场景编写异常处理逻辑。
- 如果你写了 200 行代码，实际 50 行就能搞定，必须重构精简。

自我审视：资深工程师会不会觉得这段代码设计得过于复杂？如果会，立刻简化。

## 3. 精准局部改动

**只改动必须修改的内容，只清理自身改动带来的冗余。**

编辑已有代码时：
- 不要擅自 “优化” 周边无关代码、注释和排版格式。
- 不去重构原本没有问题的代码。
- 遵循项目现有编码风格，哪怕你有自己更习惯的写法。
- 发现无关的废弃代码，只做备注说明，不要擅自删除。

当你的改动产生无用代码时：
- 移除因你的修改而变得无用的导入、变量、函数。
- 除非明确要求，否则不要删除项目原本就存在的废弃代码。

检验标准：每一行改动的代码，都能直接对应到用户的需求。

## 4. 目标导向执行

**明确验收标准，循环迭代直至验证通过。**

把任务转化为可落地验证的目标：
- “增加校验逻辑” → “先编写非法入参测试用例，再编码让用例通过”
- “修复漏洞 Bug” → “先编写可复现问题的测试用例，再编码修复并通过”
- “重构某模块” → “确保重构前后所有测试均可正常通过”

多步骤任务先列出简要执行规划：
```
1. [步骤] → 验证标准：[校验项]
2. [步骤] → 验证标准：[校验项]
3. [步骤] → 验证标准：[校验项]
```

清晰的验收标准可以让你自主闭环完成任务；模糊的标准（只要求 “能正常运行”）会需要反复沟通确认。

## 5. 前后端文件规模与拆分规范

**控制单文件复杂度，超过合理规模时按职责拆分，命名和目录结构遵循企业级工程实践。**

前端代码：
- 组件文件应尽量控制在 300 行以内。
- 如果组件文件超过 300 行，优先按职责拆分为自定义 hook、工具函数或子组件。
- 拆分后的 hook 应命名为 `useXxx`，工具函数应体现明确业务语义，子组件应使用清晰的领域命名。
- 不为了行数机械拆分，拆分边界应围绕状态逻辑、事件处理、数据转换、展示组件等稳定职责。
- 拆分后仍需保持调用链清晰，避免把一个复杂组件拆成多个无语义的小文件。

后端 Rust 代码：
- 单个 Rust 文件应尽量控制在 400 行以内。
- 如果文件超过 400 行，优先按领域职责拆分为模块、服务、类型定义、错误处理、工具函数或测试模块。
- 模块命名应表达业务边界或技术职责，避免使用 `common`、`utils` 等含义过宽的名称，除非项目已有明确惯例。
- 拆分时保持公开 API 收敛，能私有的函数、类型和模块不要暴露为 `pub`。
- 不因拆分破坏现有调用语义、错误处理方式和测试结构。

通用要求：
- 拆分前先说明拆分依据、预期文件结构和验收标准。
- 拆分必须服务于可读性、可维护性和测试便利性，不做无意义的目录膨胀。
- 新文件命名应与现有项目风格一致，并能让维护者从文件名直接理解职责。
- 拆分完成后应运行相关类型检查、构建或测试；无法运行时必须说明原因。

---

**准则生效的表现：** 代码变更里无多余改动、减少因过度设计导致的反复重写、在编码前先澄清疑问，而非出错后再补救。
